# 扫雷 v2.3 设计文档

日期：2026-07-10 · 目标版本：v2.3.0 · 基线：v2.2.0（main @ 2272d4c）

## 0. 决策摘要与范围

本版本以 v2.2.0 发布后审计和 iOS 27 真机反馈为输入。用户已逐项确认以下决策：

1. 玻璃强度选 **C（Clear / 强玻璃）**：高透明、明显色散、镜面边缘和触点跟随；
2. 覆盖范围选 **B（功能层）**：玻璃服务于控制与反馈，不覆盖棋盘内容层；
3. 修复范围选 **完整修复**：处理全部 P1/P2，以及 0 秒成绩、CSP、版本与元数据问题；
4. 动效选 **分层动效**：玻璃控件完整果冻回弹，棋盘格短促微弹，结果弹窗弹性进入；
5. PWA 更新选 **安全提示更新**：仅在首页或关卡菜单提示，由用户确认，游戏中绝不打断；
6. 技术路线选 **CSS 液态材质 + 轻量 Pointer Events 控制器**，不采用 Canvas/SVG 实时位移滤镜。

### 0.1 目标

- 让 iOS 27 上的 Liquid Glass 在第一眼即可辨认，同时保留扫雷现有的自然绿色、藤蔓和纸感，不做成无差别的系统 UI 仿品；
- 让触摸、鼠标和键盘激活都得到成套的果冻 Q 弹反馈，并在 1232 格大棋盘上保持响应；
- 消除审计确认的键盘、读屏、对比度、手机可操作性、焦点、存档、无尽曲线、PWA 更新和发布链问题；
- 产出经 Web、Windows、自动化浏览器和 iOS 27 真机验收的 v2.3.0 正式版。

### 0.2 非目标

- 不改变无猜生成器、求解器规则、50 关数表、胜负规则、音效参数或 SaveData 字段；存档继续使用 schema v3，但允许为防旧版覆盖而迁移到受保护键；
- 不新增关卡、模式、账号、云同步、震动 API 或每日挑战；
- 不把每个棋盘格、藤蔓节点或大面积内容卡片玻璃化；
- 不使用 Canvas、WebGL、SVG displacement filter 或持续运行的物理循环；
- 不在没有证书的情况下伪造 Windows 签名。v2.3 只完善签名接入与验证，安装包是否签名以实际证书状态为准；
- 不擅自授予开源许可。仓库当前没有 LICENSE，v2.3 保持 private/unlicensed 语义并移除占位元数据；开源许可需由所有者另行决定。

## 1. 设计原则与资料依据

Apple 对 Liquid Glass 的定位是覆盖在内容之上的功能层材料：它让下方颜色透出，随内容和交互变化，同时必须保持控件层级与可读性。v2.3 因此采用“强材质、克制覆盖”，而不是把整个界面做成透明玻璃。

参考资料：

- [Apple Liquid Glass 技术概览](https://developer.apple.com/documentation/TechnologyOverviews/liquid-glass)
- [Apple Human Interface Guidelines：Materials](https://developer.apple.com/design/human-interface-guidelines/materials)
- [Apple Human Interface Guidelines：Motion](https://developer.apple.com/design/human-interface-guidelines/motion)
- [WebKit：Safari backdrop-filter 支持](https://webkit.org/blog/15865/webkit-features-in-safari-18-0/)

实现遵循四条原则：

1. **功能层与内容层分离**：玻璃提示“这里可以操作”，棋盘和正文保持稳定；
2. **强效果不牺牲清晰度**：背景越复杂，玻璃衬底越实，正文对比度不依赖模糊是否生效；
3. **动效响应输入而非自行表演**：不使用常驻动画，触摸、点击、键盘或弹窗出现时才运动；
4. **能力渐进增强**：不支持玻璃、减少透明度、减少动态、强制色彩等环境都有确定回退。

## 2. Liquid Glass 材质架构

### 2.1 样式令牌

`src/ui/style.css` 建立独立的玻璃令牌层，避免每个按钮各写一套阴影和透明度。初始视觉目标如下，实施时只允许在真机校准范围内微调：

```css
--glass-blur: 28px;
--glass-saturate: 210%;
--glass-tint: rgba(255, 255, 255, .38);
--glass-tint-strong: rgba(238, 248, 239, .50);
--glass-edge: rgba(255, 255, 255, .82);
--glass-shadow: rgba(42, 57, 45, .20);
--glass-prism-a: rgba(115, 226, 196, .30);
--glass-prism-b: rgba(154, 164, 255, .25);
--glass-prism-c: rgba(255, 171, 210, .22);
```

每个玻璃控件由以下视觉层合成：

- 元素本体：透明衬底、`backdrop-filter` / `-webkit-backdrop-filter`、高亮边框和内外阴影；
- `::before`：受控的三色色散/棱镜层，只随交互偏移，不做常驻漂移；
- `::after`：以 `--glass-x`、`--glass-y` 为中心的镜面高光；
- 内容层：文字和图标始终位于光学层之上，使用独立的高对比前景色。

玻璃提供 clear、tinted、compact 三种组合，不派生页面专属版本。主操作使用绿色着色玻璃；次要操作使用 clear；空间紧张的顶部/底部栏使用 compact。

### 2.2 覆盖边界

使用强玻璃的组件：

- 首页主要操作、无尽入口、声音按钮和模式切换；
- 关卡菜单的返回/导航控件；
- 游戏顶部状态栏、底部操作栏、挖掘/插旗模式切换、重开与缩放入口；
- 胜负结果弹窗；
- PWA 新版本提示。

同一视觉簇只允许一层 `backdrop-filter`：游戏顶部/底部栏、结果面板、PWA 提示和模式切换以外层作为唯一光学表面，内部按钮只做高光、着色和回弹，不再叠加 blur；首页独立按钮各自是一层表面。静态雷数、计时和标题块不进入按压状态机。

保持内容材质的组件：

- 棋盘和所有棋盘格；
- 首页大面积信息面板；
- 关卡藤蔓、节点和说明正文；
- 普通背景装饰。

这条边界是性能与视觉层级约束，不得为了“统一”把 1232 个格子加上 `backdrop-filter`。

### 2.3 交互控制器

新增单一职责模块 `src/ui/liquid-glass.ts`。`data-liquid-glass` 标记真正的光学表面，`data-jelly` 标记接收按压回弹的交互目标；独立玻璃按钮同时具有两者，嵌套在玻璃面板内的按钮只有 `data-jelly`。控制器从交互目标找到最近的光学表面，把坐标变量写在表面、把按压/回弹类写在目标；同一光学表面一次只跟随一个主指针。变量为：

- `--glass-x` / `--glass-y`：触点在控件内的百分比坐标；
- `--glass-dx` / `--glass-dy`：以控件中心为原点的归一化方向；
- `.is-glass-pressed`、`.is-glass-releasing`：按压和回弹阶段。

控制器使用 Pointer Events，并覆盖 `pointerdown`、合并后的 `pointermove`、`pointerup`、`pointercancel`、`lostpointercapture`。同一控件一次只接受一个主指针；多指不会留下卡住的 pressed 状态。移动只在活动指针存在时通过至多一个 `requestAnimationFrame` 合并，空闲时没有帧循环。键盘激活使用控件中心作为高光位置，并走同一回弹状态机。

模块初始化一次并返回清理函数；视图替换不重复挂全局监听器。控制器不掌握路由、按钮业务或棋盘状态。

### 2.4 回退策略

- 不支持 `backdrop-filter`：使用约 92% 不透明的着色面板、清晰边框与阴影；
- 系统请求减少透明度且浏览器支持对应媒体查询：关闭背景穿透和色散，改用实色高对比面板；
- 因 [WebKit 当前仍未保证 `prefers-reduced-transparency`](https://bugs.webkit.org/show_bug.cgi?id=175497) 把 iOS“降低透明度”暴露给网页，首页次要控制区增加“降低透明度”开关（默认关闭、`aria-pressed` 表示状态），作为始终可用的手动回退；偏好写入独立的 `minesweeper-ui-prefs-v1`，不改变 SaveData；
- `prefers-contrast: more`、手动“降低透明度”或未来可用的减少透明度媒体查询任一命中时，都启用相同的实色模式；
- `forced-colors: active`：交还系统颜色，保留可见边框和焦点环；
- 背景过于复杂的游戏栏使用 `--glass-tint-strong`，不得通过降低文字对比度换取透明感。

## 3. 分层动效与输入反馈

### 3.1 玻璃控件：完整果冻回弹

`pointerdown` 必须同步写入 pressed 类和触点 CSS 变量，使测试在下一次 `requestAnimationFrame` 前即可观察到按压态。控件缩放至约 0.95–0.97，并向触点方向平移最多 2px、倾斜最多 1.2deg；高光聚拢到触点，色散向反方向轻移。

松开阶段使用约 420ms 的有限关键帧：

1. 从压缩态回到 1；
2. 越过至约 1.035–1.045；
3. 回落到约 0.992；
4. 稳定在 1。

这形成一次主回弹和一次较小的二次回正。重复快速点击必须重新开始当前控件的回弹，不堆积动画、不延迟业务点击。拖出、取消或页面切换与正常松开走同一清理路径。

### 3.2 棋盘格：短促微弹

- 直接挖掘或插旗的格子播放约 160–200ms 的微弹；
- 连锁展开从本次新揭开的格子中选择与视口相交者，再按“距触发格的曼哈顿距离、逻辑索引”升序取前 64 个播放微弹；其余格子直接进入最终状态；
- 格子不使用 blur、filter 或长期 `will-change`；
- 棋盘动作先提交状态，再播放视觉反馈，动画不阻塞计时与输入。

### 3.3 结果弹窗

遮罩先在约 120ms 内柔和压暗；面板以轻微纵向拉伸进入，并在约 440ms 内完成“放大—越过—回正”。焦点必须在面板完成可交互挂载后立即进入，不等待动画结束。

### 3.4 系统偏好

- `prefers-reduced-motion: reduce`：取消倾斜、位移、越过和批量格子动画，只保留不超过 120ms 的颜色/透明度变化；
- 减少透明度：按 §2.4 使用实色面板；
- 动效不是状态的唯一表达方式，按下、选中、胜负、更新就绪均有颜色、图标或文字反馈。

## 4. 无障碍与焦点模型

### 4.1 棋盘网格

DOM 所有权固定为 `grid > row > gridcell`：棋盘容器使用 `role="grid"`、`aria-rowcount`、`aria-colcount` 和可读名称；每个视觉行有 `role="row"` 与 `aria-rowindex`；每个逻辑格使用 `role="gridcell"`、`aria-rowindex`、`aria-colindex`。视觉转置时必须重建 row 所属关系，而不只是修改标签。

格子可读名称包含视觉行列，并严格区分“未揭开”“已插旗”“已揭开，空白”“已揭开，周围 N 颗雷”“雷”“已触发的雷”“错误旗帜”“未揭开的安全格”。例如“第 3 行，第 5 列，已揭开，周围 2 颗雷”。

采用 roving tabindex：整个棋盘在 Tab 顺序中始终只有一个格为 `tabindex="0"`，其余为 `-1`。键盘规则固定为：

- 方向键：移动一格；
- Home / End：当前视觉行首 / 行尾；
- Ctrl+Home / Ctrl+End：棋盘首格 / 末格；
- Enter / Space：挖掘或对已开数字执行 chord；
- F：切换当前格旗帜；
- 焦点移动不得触发挖掘，也不得被棋盘平移吞掉。

Enter/Space 的自定义处理必须阻止 Space 页面滚动，并抑制随后由原生 button 合成的重复 click；一次按键只能改变一次盘面。横竖屏重排后，焦点保持在同一个逻辑格；ARIA 行列更新为新的视觉位置。计时器不按秒进入 live region，避免读屏持续播报；旗数和结果通过节制的 `aria-live="polite"` 状态区通知。除 DOM/自动化断言外，至少完成一次 iOS VoiceOver 和 Windows NVDA 的棋盘冒烟检查。

### 4.2 页面与菜单焦点

- 首页主标题设置 `tabindex="-1"`，挂载后由程序聚焦；
- 关卡菜单挂载后焦点进入当前/下一可玩关卡；50 关全部完成时聚焦 L50；
- 关卡节点采用 roving tabindex：Up/Left 前移、Down/Right 后移、Home 到 L1、End 到最高已解锁关，锁定节点从键盘移动序列中跳过，Enter 启动当前关；
- 从菜单进入游戏后，焦点进入棋盘当前格；
- 使用 `replaceChildren` 切页前记录可恢复目标，切页后显式设置焦点，不允许焦点落回 `body`；
- 所有图标按钮都有可读名称，模式切换暴露选中状态。

### 4.3 结果弹窗

结果弹窗使用 `role="dialog"`、`aria-modal="true"`、固定标题 ID 和固定摘要 ID；`aria-labelledby` 指向标题，`aria-describedby` 始终指向包含胜负原因、用时、纪录与保存警告的摘要节点。弹窗作为背景游戏内容的兄弟层挂载，避免把自身置于 `inert` 子树。打开时：

- 背景主区域设置 `inert`，兼容回退时阻止背景聚焦；
- 焦点进入主要操作按钮；
- Tab / Shift+Tab 在弹窗内循环；
- Escape 关闭弹窗并显示已结束棋盘，同时移除背景 `inert`；
- 关闭后按固定优先级恢复焦点：仍存在的触发逻辑格 → 最后一个逻辑格 → 游戏主标题。

### 4.4 对比度与可触摸尺寸

- 普通文本对比度至少 4.5:1，大号文本至少 3:1；
- 控件边界、焦点环和状态图形相对邻近颜色至少 3:1；
- 首页统计、版本文字和所有难度按钮不再使用已知不达标的浅色组合；
- 非棋盘按钮的触摸区域至少 44×44 CSS px；棋盘格在“可操作尺寸”下的目标宽高至少 24 CSS px，实测容差按 §5.2 的 23.5px；
- 焦点环不能被 `overflow`、玻璃高光或缩放裁切。

大号文本严格按 WCAG 判定为至少 24 CSS px 的常规字重，或至少 18.66 CSS px 且为粗体；其余一律按 4.5:1。验收建立“前景令牌 × 允许的最亮/最暗背景 × normal/hover/pressed/disabled/focus”矩阵，计算透明层合成后的实际颜色；axe 只作补充，不能替代玻璃与渐变背景的对比度检查。

## 5. 手机布局、缩放与旋转

### 5.1 页面缩放和安全区

`index.html` viewport 改为允许系统缩放，只保留 `width=device-width, initial-scale=1, viewport-fit=cover`，静态测试禁止出现 `user-scalable=no` 或 `maximum-scale`。顶部和底部栏继续使用 `env(safe-area-inset-*)`，但在 320×568 下把根字号放大到 200% 时仍可换行；除可平移棋盘外，页面必须满足 `scrollWidth <= clientWidth`，控件不裁切且非棋盘按钮仍至少 44px。

### 5.2 两种棋盘视图

视口模块明确区分：

- **适合屏幕**：完整展示棋盘，允许格子小于 24px，只用于观察全局；
- **可操作尺寸**：目标格边长至少 24 CSS px；考虑子像素取整，浏览器验收要求 `getBoundingClientRect()` 的宽、高均 ≥23.5px，并允许平移查看超出区域。

若 `fitScale` 会让格子小于 24px，首次进入该设备上的超大棋盘时默认使用“可操作尺寸”，以首个可玩/当前焦点格为中心，并显示一次“单指拖动棋盘，双指缩放棋盘”的非模态提示。提示记录为独立 UI 偏好，不修改 SaveData schema。游戏栏提供“适合屏幕”和“可操作尺寸”两个有可读名称的入口。

棋盘手势固定为：轻点执行当前触摸模式；350ms 长按切换旗帜；单指移动达到 `TOUCH_SLOP_PX = 10` 后只平移、不再触发格子；双指只缩放棋盘局部视口。`touch-action: none` 只作用于棋盘交互区，首页、菜单和悬浮控件继续允许浏览器页面缩放；iOS 人工验收还需确认 Safari 页面缩放与棋盘局部缩放两条路径都存在。

平移钳制继续允许棋盘进入悬浮栏下方，但当前焦点格通过键盘移动后必须自动滚入净空可见区。

### 5.3 旋转重排

宽屏判断不再只在 `renderGame` 时计算。`resize` / orientation 变化时：

1. 结束当前棋盘 pointer capture、长按计时和临时按压态；
2. 捕获当前逻辑焦点格和视口中心对应的逻辑格；
3. 重新计算是否转置、视觉行列、棋盘像素尺寸和 insets；
4. 重建或重排视觉映射；
5. 优先保持焦点格可见，其次保持中心逻辑格位置；
6. 重新应用平移钳制和 ARIA 行列。

旋转必须保持同一个逻辑焦点格；中心逻辑格的位置误差不超过一个视觉格。用户缩放按“旋转前的实际格边 CSS px”保持，随后仅为满足新视口合法范围而钳制。纵→横→纵过程中不得重置盘面、计时、旗帜、当前模式；计时继续递增而不归零。

## 6. 数据可靠性与规则修复

### 6.1 防降级覆盖

防护分为安装器和数据两层，因为 v2.3 的配置无法反向修改已经发布的 v2.1/v2.2 安装器：

- v2.3 及后续 NSIS 配置显式设置 `allowDowngrades: false`，使从本版本开始生成的安装器拒绝覆盖已安装的更高版本；
- 新增受保护存档键 `minesweeper-save-v3`，现有 `minesweeper-save-v1` 改称 legacy key；
- v2.3 优先且只持续读写受保护键；受保护键不存在时才从 legacy key 读取、执行现有 schema 迁移/校验，并立即把结果写入受保护键；
- 受保护键一旦存在，后续即使旧程序覆盖 legacy key，v2.3 也不得回退读取 legacy key；
- legacy key 不删除，避免主动破坏用户临时运行旧版时仍可读取的旧快照，但 v2.3 的新进度不会再写回该键。

验收必须模拟“首次迁移成功 → 旧版把 legacy key 覆盖成默认值 → 再次启动 v2.3”，确认受保护键中的关卡、成绩、声音和无尽进度全部不变。若用户实际运行旧安装器，它仍可能由旧安装器自身允许降级，但不能再破坏 v2.3 的受保护数据。

### 6.2 无尽进度保存失败

存储写入结果不得再被 `main.ts` 硬编码为成功。无尽胜负采用以下数据流：

```text
当前会话状态 → 计算下一状态 → 尝试保存完整最新快照
                              ├─ 成功：清除待重试状态，标记 persisted=true
                              └─ 失败：保留会话状态与最新待写快照，标记 persisted=false
```

- UI 始终按本次会话的真实胜负继续，不因写盘失败回滚；
- 写盘失败显示非阻塞、可读屏的“进度暂未保存，将自动重试”；
- 待重试快照只保留最新一份，并在下一次成绩/无尽/声音状态写入、进入首页或关卡菜单、以及页面转入 `visibilityState === "hidden"` 时重试；成功后清除警告；
- 关闭应用前仍失败时只能明确告知风险，不能声称已持久化；
- `recordEndlessWin` 返回 `{ streak, bestStreak, newBest, persisted }`；`recordEndlessLoss` 返回 `{ streak: 0, bestStreak, persisted }`。调用者不得自行伪造 `persisted`。

`storage.ts` 内的内存 `SaveData` 是唯一权威会话状态；所有写路径（普通成绩、无尽胜负、声音偏好）同步串行地更新它。失败时 `pending` 只保存最新完整快照，后来的状态覆盖较旧 pending，绝不回放过期写入。公开契约增加 `flushPending(): "idle" | "saved" | "failed"`；路由和 `visibilitychange` 只调用该契约，不直接接触 backend。普通关卡失败也显示同一全局保存警告，成功 flush 后所有页面清除警告。本版本不引入 IndexedDB 或服务端存储。

回归序列固定包含“无尽胜利写失败 → 下一局失败写失败 → 返回首页写成功”，最终落盘必须是最新的 streak=0 和已更新 bestStreak，而不是任一中间快照。

### 6.3 无尽曲线不变式

`endlessSpec(streak, rng)` 继续保持确定性和同连胜阶段的形状变化，但随机性不能改变难度顺序：

- `0 ≤ streak < 15`：生产代码导出只读 `ENDLESS_SHAPES_BY_STREAK`，每个 streak 至少包含两个合法 `(width,height)` 候选，且候选尺寸带互不倒退；任意下一 streak 候选的格数和雷数都不小于任意上一 streak 候选；
- `streak ≥ 15`：规格固定为 L50 的 28×44、285 雷和 2400 秒，不再随机抖动；
- 所有未封顶候选满足 `1.4 ≤ height / width ≤ 1.6`、`width < height`、格数 ≤1232、雷数 ≤L50；
- nominal density 定义为 `22.1 + (23.2 - 22.1) * min(streak, 15) / 15`（百分数），整数雷数统一为 `floor(width * height * nominalDensity / 100)`；封顶值以 L50 的 285 雷为准，实际密度为 `mines / (width * height)`；
- 随机数只用于从该 streak 的显式候选数组中选索引，不再先抖动再被动截断。

测试直接枚举生产导出的全部候选，并对相邻阶段做笛卡尔积比较；另以 streak 0–60 和边界随机值（至少 0、0.5、接近 1）验证选择器及封顶。只测“固定 rng 下单调”不再算通过。

### 6.4 0 秒成绩

最佳时间的合法范围统一为“有限数且 ≥0”。写入、读取、迁移和首页显示必须一致接受 0；负数、NaN、Infinity 和错误类型继续丢弃。`recordWin(1, 0)` 保存后重新加载仍为 0，并能触发相应解锁/最佳成绩逻辑；首页明确显示 `0:00`，所有存在性判断使用 `value !== undefined/null` 或等价方式，不得依赖 truthy。

## 7. PWA 安全提示更新

### 7.1 状态模型

Vite PWA 固定使用 `registerType: "prompt"` 和显式 `virtual:pwa-register`，不再组合 `autoUpdate` 与无法触发自动接管的配置。新增独立的 `src/ui/pwa-update.ts` 协调器，状态为：

- `idle`：没有等待版本；
- `ready`：新 Service Worker 已下载并等待；
- `deferred`：用户选择稍后；
- `activating`：用户确认，正在调用 `updateSW(true)`；
- `error`：激活失败，继续运行旧版本并允许重试。

协调器保存更新函数和状态，不直接持有某一页的 DOM。

### 7.2 路由规则

- 安全页面仅为首页和关卡菜单；游戏页和结果覆盖层都不是安全页面；
- 游戏中收到 `onNeedRefresh` 只进入 `ready`，不弹窗、不抢焦点、不刷新；
- 下一次进入首页/菜单时挂载非模态玻璃提示“新版本已准备好”，提供“立即更新”“稍后”；
- “稍后”把状态设为 `deferred`，当前安全页面内不重复骚扰；离开该安全页面时若更新仍在等待则恢复 `ready`，再次进入或下次启动时重新提示；
- “立即更新”只调用一次注册函数返回的 `updateSW(true)`，由 vite-plugin-pwa 完成 skip-waiting 和页面刷新；协调器不得再监听 controllerchange 后手动 `location.reload`，避免双刷新；失败时显示“更新未完成，请稍后重试”，不进入刷新循环；
- 更新提示不自动夺取焦点，但通过状态区可被读屏发现。

离线启动、预缓存和旧版本正常运行能力保持不变。已打开的 v2.2 客户端没有本提示协调器，因此升级到 v2.3 时不强制接管；关闭全部旧客户端后等待中的 v2.3 Service Worker 可正常激活。上述安全提示从 v2.3 之后的更新开始完整生效。除状态机单测外，验收必须用相同 scope 的两个 production build 做 A→B Service Worker 集成测试，观察真实 waiting、游戏中延迟、安全页提示和一次刷新接管，不能只 mock `onNeedRefresh`。

实现语义以 [vite-plugin-pwa 的 prompt 更新流程](https://vite-pwa-org.netlify.app/guide/prompt-for-update.html) 为准。

## 8. 安全、版本与发布元数据

### 8.1 Tauri CSP

桌面端改用以下最小权限起始策略：

```text
default-src 'self';
script-src 'self';
style-src 'self' 'unsafe-inline';
img-src 'self' asset: http://asset.localhost data: blob:;
font-src 'self' data:;
connect-src ipc: http://ipc.localhost;
object-src 'none';
base-uri 'none';
frame-ancestors 'none'
```

只允许在打包应用的实际资源被该策略拦截且有明确用途时增加来源；不得使用 `*`、`unsafe-eval`、远程 script 源或退回 `null`。静态测试解析配置并逐 directive 断言上述允许列表，同时断言任意 `https://example.com` script/connect 来源不被策略允许；打包启动和完整主流程中控制台不能出现 CSP 误拦截。

### 8.2 版本权威源与一致性检查

- `package.json` 是版本权威源；版本脚本把它同步到 `src-tauri/tauri.conf.json` 和 `src-tauri/Cargo.toml`，三处在发布 commit 中均为 2.3.0；
- 界面 `APP_VERSION` 从构建期 package 版本注入，不再手工复制；
- 普通测试/CI 模式只校验三处版本一致；tag CI 模式额外校验 `GITHUB_REF_NAME` 必须等于 `v${package.version}`；
- Cargo 元数据固定为 `name = "minesweeper"`、`description = "A no-guess Minesweeper game with 50 levels and endless mode"`、`authors = ["whtanghaonan-wang"]`、`repository = "https://github.com/whtanghaonan-wang/minesweeper"`、`publish = false`；
- npm 同步写入上述 description、author 和 repository；
- npm 保持 `private: true` 并明确 `license: "UNLICENSED"`；Cargo 移除 `A Tauri App`、`you` 和空 license/repository 占位值，不声明不存在的开源许可证。

### 8.3 Windows 签名

- workflow 的证书契约固定为 `WINDOWS_CERTIFICATE`（base64 PFX）、`WINDOWS_CERTIFICATE_PASSWORD` 和由证书颁发方提供的 `WINDOWS_TIMESTAMP_URL`；不得与 Tauri updater 私钥混用；
- 三项 secret 必须“全部存在或全部缺失”；只配置一部分时 workflow 直接失败，避免静默降级为未签名；
- 三项 secret 都存在时，把 PFX 临时导入 `Cert:\CurrentUser\My`，读取导入证书 thumbprint，并通过临时 Windows 配置向 Tauri 提供 `certificateThumbprint`、`digestAlgorithm: "sha256"` 和 timestamp URL；日志不得输出证书内容或密码，结束后清理临时文件；
- 有证书时签名步骤必须成功，主程序 EXE 和 NSIS 安装包的 `Get-AuthenticodeSignature` 都必须为 `Valid`；
- 没有证书时允许产出安装包，但两者必须实测为 `NotSigned`，Release 明确标注“未签名”，同时输出实际签名状态和 SHA-256；
- 不把“已配置签名入口”描述成“安装包已签名”。

证书接入遵循 [Tauri Windows Code Signing](https://v2.tauri.app/distribute/sign/windows/) 的 PFX 导入与 thumbprint 配置流程。

## 10. CI 与 v2.3.0 发布流程

### 10.1 CI

- Pages workflow 顺序固定为 `npm ci` → 完整测试 → `build:web` → 部署；测试失败不得发布；
- Windows workflow 同时支持指定 commit 的 `workflow_dispatch` 候选构建和 `v*` 标签构建；两条路径都在干净 runner 上运行测试、构建 NSIS、检查签名状态、计算 SHA-256 并上传制品，标签路径额外校验标签与三处版本完全一致；
- workflow 使用最小 GitHub token 权限，依赖 lockfile，发布日志记录 commit SHA、Node/Rust 版本和校验值。

本项目把 main 的 Pages 部署作为未打标签的 Web 候选通道，把 `v2.3.0` + GitHub Release 作为正式版标记；因此 main 候选可能先于正式标签在线，但必须通过 Pages CI，且真机验收期间 main 锁定在同一 SHA。

### 10.2 发布闸门

只有以下条件全部满足才创建正式标签：

1. 单元测试、类型检查、Web 构建、桌面构建通过；
2. Chromium/WebKit 主流程、无障碍和手机尺寸验收通过；
3. 指向待发布 commit 的 Windows 候选 workflow 成功，安装包签名状态和 SHA-256 已记录；
4. 工作树干净，版本均为 2.3.0；待发布 commit 推送到 main 后 Pages CI 成功，并作为未打标签的 Web 候选提供真机验收；
5. 相同 commit 的 Pages 候选与 Windows 候选通过 iOS 27 / Windows 真机验收，main 在验收期间未前移。

随后：在已通过真机验收的同一 commit 上创建带注释的 `v2.3.0` 标签 → 等待标签 Windows workflow 复现成功 → 创建/核对 GitHub Release → 附加 NSIS 安装包与 SHA-256 → 对线上 Pages 做一次安装后冒烟检查。标签 workflow 失败时不得创建 GitHub Release；可重试的基础设施故障只重跑同一 SHA，若必须改代码则先删除尚未发布的本地/远端 `v2.3.0` 标签，产生新候选并重新走完整闸门，绝不把失败标签保留为正式版。

## 12. 防回退决策

- 强玻璃只属于功能层；棋盘格和大内容面板不得玻璃化，同一视觉簇不得嵌套 backdrop-filter；
- 玻璃必须有触点跟随、主回弹和二次回正，但空闲时不得运行持续动画；
- Enter/Space 固定为挖掘，F 固定为插旗；触摸模式切换不改变键盘快捷键语义；
- 超大棋盘默认可操作性优先于“一屏看全”，同时保留一键全局查看；
- 无尽随机性只能改变同阶段形状，不能让下一 streak 的格数或雷数下降；streak 15 起必须完全恒定；
- 写盘失败可以保留本次会话进度，但必须明确未持久化并重试，绝不伪造成功；
- v2.3 只持续读写受保护存档键，旧版 legacy key 后续变化不得覆盖受保护数据；
- PWA 只在首页/菜单提示，且只有用户确认才激活刷新；
- 没有证书就如实发布未签名安装包，不把配置完成写成签名完成；
- v2.3.0 正式标签必须在自动化、浏览器和 iOS 27 真机验收全部通过后创建。
