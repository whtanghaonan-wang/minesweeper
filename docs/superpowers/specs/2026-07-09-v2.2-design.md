# 扫雷 v2.2 设计文档

日期:2026-07-09 · 目标版本:v2.2.0 · 基线:v2.1.0(main @ 74ec889)

## 0. 范围

五项工作(源自 roadmap 两条承诺 + 用户本次新增两条 + 音效配套),均已由用户逐项拍板:

1. **生成器修补式重构**(甲·修补式生成 + 乙·求解器提速的便宜部分),使 28×44=1232 格大盘过 500ms 生成闸门;
2. **关卡表二次调整**:21-50 关恢复高瘦大盘递进,终焉档封顶 28×44(兑现 v2.2-roadmap 用户签核承诺);
3. **无尽模式**:通关 50 关解锁,递进式连胜,首页入口;
4. **全屏棋盘 + 液态玻璃悬浮菜单**(所有端统一,对齐 iOS 27 Liquid Glass 质感);
5. **插旗音效**(甲·木钉入座,插/拔都响、拔旗低一档)。

顺带小修(一行级,随实现顺路清理):首页"⏱ 最快"措辞改"⏱ 最近通关"(v2.1 终审 Minor#4);滚轮缩放时置空按下格缓存 `downCellVi`(v2.1 终审 Minor#3)。

**非目标**:不新增求解器推理规则(会漂移"无猜"定义域,连 1-20 关可行性都会被重新定义);不做 Web Worker 预生成(闸门达标后无必要);键盘操作、每日挑战仍无排期。1-20 关字节冻结不动。

## 1. 生成器修补式重构

### 1.1 现状与瓶颈

`generate()` 现为"随机撒雷 → `isSolvable` 全程推演 → 不可解则整盘作废重来"。盘越大,任意角落出现死结的概率越高,重试次数指数级增长:28×44 实测最坏 3122ms,远超 500ms 闸门,v2.1 被迫把关卡表压到 950 格封顶。

### 1.2 修补式循环(甲)

`generate(level, firstIdx, rng, onSlow?)` 签名不变,内部改为:

```
撒雷(首格 3×3 安全区外)                        ← 同现在
loop:
  从 firstIdx 用现行求解器全程推演
  推完 → 返回该盘                               ← 本次推演即无猜证明,无需另行终验
  卡死 → 修补:
    把全部雷收起,只在「本轮尚未揭开的格子」里重撒   ← 已推开安全骨架保持无雷
    推理旗标清零
    修补计数 +1;达上限 PERTURB_MAX(50)→ 整盘作废,重新撒雷从头来
```

关键性质:

- **返回的盘必定完整通过了与现行一字不差的 `isSolvable` 推演**——无猜保证、健全性哨兵(误标雷/误开雷即 throw)、"绝不降级"全部原样;
- 效率来源:每次修补把撒雷候选域缩小到未开区,安全骨架滚雪球式扩大,收敛远快于全盘重试;
- 修补后必须**从首格从头重推**(修补改变已开数字格的邻雷数,旧推理进度不可续用);
- 确定性:整个过程只消费传入的 `rng` 流,同种子同结果,GEN_BENCH 可复现;
- `onSlow` 回调语义保留(按累计尝试规模触发)。

### 1.3 求解器提速(乙,只做不改语义的)

- 热路径数据结构:`isSolvable` 内部用类型数组(`Uint8Array`)替代 boolean 数组,削减 `Set` 高频建销;
- 子集推理对的早剪枝(长度预筛、排序复用);
- **禁止新增/删除推理规则,推理层级(基础→子集→全局终局→边界穷举+计数)不变**;
- 以基准剖析为准,不为提速而重构 `board.ts` 等公共模块语义。

### 1.4 验收

- GEN_BENCH 闸门(不变的标准):新 21-50 表全关 5 种子单盘最坏 <500ms,含 28×44 终焉盘;
- 正确性单测:修补只落未揭开格(已开骨架永不埋雷)、修补上限触发整盘重来、返回盘复跑 `isSolvable` 必真、同种子确定性;
- 现有 generator/solver 测试全绿(语义未变的直接证据)。

## 2. 关卡表二次调整(21-50)

- 恢复**高瘦递进**:全部 height > width,宽高比约 1:1.4~1.6;
- 终焉档封顶 **28×44=1232 格**;起点衔接 L20(20×34),格数/雷数逐关**严格递增**;
- 密度带沿用 **[22.1%, 23.2%]** 档均值缓升;
- 限时 930s 起每关 +30s 至 1800s(不变);
- 具体数表由实现任务产出,受三重约束:`levels.test` 不变式(递增/密度带/档结构)+ GEN_BENCH 闸门 + 1-20 字节冻结;
- 档名、藤蔓选关、首页进度条等均按 `LEVELS` 数据驱动,无需改动。

## 3. 无尽模式

### 3.1 解锁与入口

- 解锁条件:`bestTimes[50]` 存在(通关第 50 关);
- 首页操作区加第三按钮"♾ 无尽";未解锁时灰态 + 🔒,副文案"通关 50 关解锁"(让玩家看见奔头);
- 解锁后首页统计行追加"♾ 最长连胜 N"(bestStreak>0 时)。

### 3.2 递进曲线(`src/core/endless.ts` 纯函数)

`endlessSpec(streak, rng): LevelSpec`:

- 格数:600 起,每胜约 +45,**封顶 1232**(约第 15 胜到顶);
- 密度:22.1% 起线性缓升,**封顶 23.2%**(与格数同步到顶);
- 形状:高瘦(1:1.4~1.6),宽度 ±1 列随机抖动(消费 rng,同档也盘盘不同);
- 限时:900s 起每胜 +100s,**封顶 2400s**;
- 设计不变式(配测试):任意 streak 的(格数, 密度)≤ 新 L50(1232, 23.2%)——GEN_BENCH 对 campaign 顶端的保障自动覆盖无尽全程;曲线单调不减;封顶后恒定;
- `tier` 取新增枚举值 `"endless"`(`TIER_NAMES` 补"无尽"、CSS 补主题色,`Record<Tier,…>` 穷尽性由编译器强制);`id` 固定 0,不参与 `bestTimes`/解锁。

### 3.3 连胜规则(宽松版,用户拍板)

- 胜一盘 → `streak+1`,`bestStreak` 同步取 max;
- 踩雷或超时 → `streak` 归零,下一盘回起步盘;
- **中途重开或退出 → 本局不计、连胜保留**(与选关模式重开习惯一致,休闲定位);
- 连胜跨会话持久化,关掉 app 不清零。

### 3.4 存档 v3(`storage.ts`)

- `SaveData` 升 `version: 3`,新增 `endless: { streak: number; bestStreak: number }`;
- **v2→v3 迁移**:`unlockedLevel`/`soundOn`/1-20 关 `bestTimes` 原样保留;**21-50 关成绩丢弃**(盘面规格全变、不可比,沿 v1→v2 先例,用户知情确认);`endless` 默认 `{0,0}`;
- 损坏字段逐项校验回退默认,与现有字段同规格;`SAVE_KEY` 不变;
- 新 API:`recordEndlessWin(): { streak, bestStreak, newBest }`、`recordEndlessLoss(): void`(归零并持久化)。

### 3.5 路由与界面

- `main.ts` 增 `gotoEndless()`:按当前 `streak` 生成盘并进入游戏页;
- `game.ts` `GameDeps` 增模式标识 `mode: { kind: "campaign" } | { kind: "endless"; streak: number }`(缺省 campaign,现有调用不受影响);无尽局标题位显示"♾ 无尽"+徽章"连胜 N",其余(雷数/时间/音效/模式切换/重开/悬浮布局)与普通局完全一致;
- 结算弹窗(`result.ts`):胜 →"🔥 连胜 N!"+主按钮【下一盘】,破纪录加徽章"新纪录·最长连胜 N";负 →"连胜止于 N"+主按钮【再来一盘】(回起步盘);次按钮【回首页】;
- 无尽局的胜负记录走 3.4 新 API,不触碰 `recordWin`;结算暂停窗口的冲刷语义(胜冲刷/负丢弃)对无尽同样成立——冲刷时照常记连胜。

## 4. 全屏棋盘 + 液态玻璃悬浮菜单(所有端)

### 4.1 布局

- `.board-viewport` 铺满整个游戏页(`position: absolute; inset: 0`,含上下菜单背后区域);
- `.game-top`/`.game-bottom` 改绝对定位悬浮层(z 上层),始终可点;`pc-hint` 并入底部悬浮区(仍仅 `pointer: fine` 显示);
- 手势归属天然正确:指针落在悬浮按钮上归按钮,落在棋盘露出区归棋盘,无需特殊处理。

### 4.2 视口数学(`viewport.ts`,配新测试)

- `Metrics` 增加上下净空 `insetTop`/`insetBottom`(游戏页实测两栏高度传入;jsdom 下为 0,自动回退现行为,测试保持确定性);
- **适配与居中用净空区**:`fitScale` 按(viewH − insets)计算,默认视图棋盘完整可见、不被菜单遮挡——未放大时体验与现在完全一致;
- **钳制界用全屏区**:轴向上盘尺寸超出净空区后,允许延伸到全屏边缘(菜单底下);盘小于净空区的轴向仍在净空区内居中。

### 4.3 液态玻璃质感(用户在 mockup 拍板)

悬浮条 pill/模式切换按钮统一:

```css
border-radius: 999px;                              /* 胶囊 */
background: linear-gradient(135deg, rgba(255,255,255,.62), rgba(255,255,255,.34));
backdrop-filter: blur(18px) saturate(185%);        /* + -webkit- 前缀 */
border: 1px solid rgba(255,255,255,.75);
box-shadow: inset 0 1px 0 rgba(255,255,255,.95),   /* 镜面高光 */
            inset 0 -1px 1px rgba(255,255,255,.28),
            0 4px 18px rgba(74,69,62,.16);
```

- `@supports not (backdrop-filter: blur(1px))` 回退 `rgba(255,255,255,.9)` 实心;
- **全局统一**:首页/选关页的按钮类控件(`.pill`、`.home-play`/`.home-select`、`.sound-btn`、返回钮等)同步换液态玻璃;有色主按钮(如绿色"开始游戏")用**着色玻璃**——色相不变,叠加同款高光/模糊质感,不降低 CTA 显著性;**大面积卡片保持实心白**(棋盘白底、`.home-panel`、结算 `.modal`、藤蔓节点)——保证可读性与层次;
- 纯 CSS 实现,零 JS、零性能包袱;active 缩放反馈沿用。

## 5. 插旗音效(甲·木钉入座,试听拍板,禁改)

### 5.1 参数(`audio.ts` 新增,音效家族五→七种)

```ts
/** 插旗 */
export function playFlag(): void {
  tone(988, "sine", 0.055, 0.2);
  tone(1976, "sine", 0.03, 0.05);
  noise(0.02, 0.07, 0, 1500);
}
/** 拔旗(低一档) */
export function playUnflag(): void {
  tone(784, "sine", 0.055, 0.18);
  tone(1568, "sine", 0.03, 0.04);
  noise(0.02, 0.05, 0, 1100);
}
```

文件头"试听拍板、修改前必须重新试听确认"的禁改注释同步涵盖两个新音。

### 5.2 接线(`game.ts`)

- `act()` 两处旗分支(开局前预旗、盘上插旗)在切换后按**新状态**响:置旗 → `playFlag()`,取消 → `playUnflag()`;
- 预旗、触屏长按、旗模式点按、PC 右键四条路径天然汇聚于这两处分支,一处接线全覆盖;
- 已开数字格点按(chord)不响旗音;长按振动反馈保留;静音短路在 audio 函数内部,照旧。

## 7. 既定决策备忘(防回退)

- 修补式生成**必须从首格从头重推**,不得续用修补前的推理进度(修补会改变已开数字格的数字,续推会破坏无猜证明链);
- 求解器推理规则集冻结(§1.3);
- 拔旗音低一档、重开不断连胜、21-50 旧成绩迁移即弃,均为用户拍板;
- 液态玻璃只上按钮类控件,大卡片保持实心白(§4.3)。
