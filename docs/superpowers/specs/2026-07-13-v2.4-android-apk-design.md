# v2.4 Android APK 设计

## 目标

为扫雷 v2.4 增加可长期更新的 Android 正式签名 APK。Android 用户先通过同签名候选包进行实机验证；验证通过后，使用同一提交创建统一的 `v2.4.0` GitHub Release。

## 已确认的产品决策

- 使用现有 Tauri 2 工程的原生 Android 目标，不新增 TWA 或 Capacitor 壳。
- 应用标识保持 `io.github.whtanghaonan.minesweeper`。
- 最低系统版本为 Android 7.0（API 24）。
- 交付一个通用 APK，包含 `aarch64`、`armv7`、`x86_64`，供用户下载单一文件安装。
- 版本继续统一为 `2.4.0`；Android `versionCode` 使用公式 `major * 1,000,000 + minor * 1,000 + patch`，因此 v2.4.0 固定为 `2004000`，后续同签名版本可覆盖升级。
- iOS 继续通过 Pages/PWA 使用；不生成 IPA。
- 正式 Release 同时提供 Android APK、Windows 安装版与便携版、全部 SHA-256 校验值，以及 Pages/PWA 使用说明。Windows 产物继续明确未配置 Windows 代码签名。

## 工程结构

1. 使用 `npx tauri android init` 在 `src-tauri/gen/android` 生成并提交 Android Gradle 工程。
2. 新增 Android 平台配置文件，明确 API 24、确定性 `versionCode`、应用名称、图标和发布构建所需配置。
3. 保持现有前端、游戏逻辑、PWA 和 Windows 配置不变；Android 使用 Tauri WebView 加载同一套本地打包前端资源，离线可启动。
4. 忽略 Android 本地构建产物、Gradle 缓存、临时 keystore 与本机环境文件，绝不将密钥或密码提交到 Git。

## 签名与密钥

- 创建专用 Android 发布 keystore，固定别名 `minesweeper`，使用强随机 store/key 密码。
- 本机备份目录固定为 `D:\minesweeper-android-signing\`，并限制为当前 Windows 用户访问。
- 该目录包含 keystore、证书 SHA-256 指纹和恢复说明；恢复密码文件同样受目录权限保护。
- GitHub Actions 使用环境 Secrets：
  - `ANDROID_KEYSTORE_BASE64`
  - `ANDROID_KEYSTORE_PASSWORD`
  - `ANDROID_KEY_ALIAS`
  - `ANDROID_KEY_PASSWORD`
- CI 仅在临时运行器内还原 keystore，构建结束后不保留；日志不得输出秘密或完整密钥内容。
- 丢失此 keystore 将无法以同一包名覆盖更新已安装的 APK，因此本机备份必须长期保留。

## 候选与正式发布流程

### 候选包

1. Android 工作流接受精确的 40 位候选 SHA。
2. 工作流验证该 SHA 等于 `origin/main` 的当前提交，防止签错旧代码。
3. 工作流运行版本检查、单元测试、Android 构建和 APK 验证。
4. 生成正式签名通用 APK、SHA-256 校验文件和构建元数据，并作为 14 天候选 Artifact 上传。
5. 用户下载候选 APK，在 Android 手机完成实机验收；此阶段不创建 GitHub Release，也不打 `v2.4.0` 标签。

### 正式发布

1. 用户确认 Android 实机验证通过后，在同一候选 SHA 创建 `v2.4.0` 标签。
2. Windows 与 Android 工作流分别构建标签产物。
3. 发布步骤下载两个平台产物，复核版本、签名状态、证书指纹和 SHA-256。
4. 创建统一 GitHub Release，上传：
   - `minesweeper-v2.4.0-universal.apk`
   - Windows 安装版与便携版
   - 所有平台的 SHA-256 校验文件和构建元数据
5. Release 说明写明 Android 7+、侧载安装步骤、Play Protect/未知来源提示、Windows 未签名提示及 Pages/PWA 地址。

## 构建与验证合同

- 本机构建环境：JDK、Android SDK、NDK、Android Rust targets 和 Tauri CLI 版本必须被记录并检查。当前机器未安装 JDK、Android Studio/SDK/NDK 或 Android Rust targets，实施阶段需要先配置。
- CI 使用固定的 JDK、Android SDK/NDK 与 Rust target 配置，避免依赖开发机状态。
- 构建命令生成 release APK，不生成 debug APK 作为发布候选。
- APK 验证至少包括：
  - 包名 `io.github.whtanghaonan.minesweeper`。
- 版本名 `2.4.0` 与 `versionCode=2004000`。
  - 有效签名和匹配的证书指纹。
  - `aarch64`、`armv7`、`x86_64` ABI 均存在。
  - APK、校验文件和构建元数据的文件名和摘要匹配。
- 任一签名、版本、ABI、测试或校验步骤失败时，工作流不得上传可发布候选或创建 Release。

## Android 实机验收

- Android 7.0 及以上设备可安装、启动和离线重启。
- 关卡进度、声音和透明度偏好持久保存。
- 首页液化玻璃点击、快速切换、连续边缘拖拽和外层反馈正常。
- 游戏棋盘、顶部胶囊、横竖屏及系统安全区域不溢出。
- 系统返回键保持默认退出行为；游戏内导航继续使用现有界面返回按钮，不引入新的复杂 Android 导航栈。
- 不请求相机、定位、通讯录等危险权限。
- Release APK 文件名为 `minesweeper-v2.4.0-universal.apk`，并提供 SHA-256 与签名证书指纹。

## 非目标

- 不发布到 Google Play，不生成 Play Store 专用 AAB。
- 不生成 iOS IPA。
- 不改变现有 Pages、PWA、Windows 包或游戏规则。
- 不在 Android 候选验证前创建或移动 `v2.4.0` 标签。
