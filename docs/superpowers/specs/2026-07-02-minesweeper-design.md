# 扫雷 DIY 设计文档

日期：2026-07-02
状态：已确认

## 1. 目标

自用扫雷游戏，一套代码发布两端：Windows 电脑独立程序 + iPhone 主屏幕应用（PWA）。核心诉求：

1. **严格无猜**：每一局的所有雷都能通过逻辑推理确定，不存在必须靠猜的局面。
2. **10 关 3 档**：简单（1–3 关）、挑战（4–7 关）、困难（8–10 关）；档间难度质变，档内按比例递增；难度由盘面大小、雷数（密度）、倒计时限时共同构成。
3. **柔和中性 UI**：大圆角、无棱角、布局协调不拥挤。
4. **基础功能**：每局倒计时、通关判定、每关最好成绩记录、逐关解锁。

## 2. 平台与技术栈

| 项 | 决策 |
|---|---|
| 核心技术栈 | TypeScript + Vite，无框架，纯 DOM 渲染（棋盘 CSS Grid，最大 288 格） |
| 电脑版 | Tauri 2 打包 Windows .exe（依赖 Rust 工具链构建；Win11 自带 WebView2 运行时） |
| 手机版 | PWA（manifest + Service Worker 离线缓存），iPhone Safari「添加到主屏幕」 |
| 部署 | GitHub Pages（免费 HTTPS；PWA 安装后离线可玩，仅首次加载和更新需联网） |
| 测试 | Vitest 单元测试（核心逻辑层） |

### 模块划分

```
core/board.ts      棋盘状态、挖开/插旗/连锁展开/胜负判定（纯逻辑，无 DOM）
core/solver.ts     无猜求解器（验证棋盘可全程逻辑推完）
core/generator.ts  布雷生成器（生成 → 求解验证 → 重试）
core/levels.ts     10 关配置表
core/storage.ts    localStorage 存档（解锁进度、最好成绩）
ui/                选关页、游戏页、结算弹窗
main.ts            装配入口
```

逻辑层（core/）与界面层（ui/）完全分离；core 不引用任何 DOM API，可独立单测。

## 3. 无猜生成算法

采用「生成时求解验证」方案：

1. **先点后布雷**：玩家点击第一格后才布雷；首格及其周围 8 格保证无雷，开局必有连锁展开。
2. **求解器模拟推理**（从首格展开后的局面开始，循环应用直到推完或卡住）：
   - **基础规则**：某数字格周围未开格数 = 其剩余雷数 → 未开格全为雷；某数字格周围已标雷数 = 其数字 → 其余未开格全部安全。
   - **子集推理**：约束 A 的未知格集合 ⊆ 约束 B 的未知格集合时，B−A 部分的雷数 = B 雷数 − A 雷数；若等于 0 则全安全，若等于格数则全为雷（覆盖经典 1-2-1、1-2-2-1 型局面）。
   - **边界穷举**：对边界未知格的连通块（大小上限初设 24 格，超限视为卡住并重新布雷）枚举所有满足约束且总雷数可行的布局，取所有布局的交集——在全部可行布局中恒为雷的格标雷、恒安全的格挖开。
3. **推不完就重新布雷**：求解器卡住则丢弃本次布雷重新生成，直到整盘可纯逻辑推完。
4. **绝不降级**：任何情况下不把需要猜的盘面交给玩家。生成设尝试上限仅用于日志报警，超限后继续尝试并显示轻量加载提示。
5. **性能后手**：预计各关生成耗时 < 1 秒；若高关卡实测卡顿，将生成迁入 Web Worker（初版不做，避免过度设计）。

## 4. 关卡体系

难度三要素：盘面、雷密度、限时（倒计时，超时判负）。档间质变由雷密度跳档体现（10–13% → 15–18% → 19–21%），档内按固定比例递增。棋盘最宽 12 列，保证 iPhone 竖屏格子 ≥ 30px；宽屏（电脑/横屏）自动交换行列为横向布局（同格数同雷数，仅呈现方向不同）。

| 关卡 | 档次 | 盘面(宽×高) | 雷数 | 密度 | 限时 |
|---|---|---|---|---|---|
| 1 | 简单 | 8×8 | 7 | 10.9% | 3:00 |
| 2 | 简单 | 9×10 | 11 | 12.2% | 3:00 |
| 3 | 简单 | 9×12 | 14 | 13.0% | 3:30 |
| 4 | 挑战 | 10×14 | 21 | 15.0% | 4:00 |
| 5 | 挑战 | 10×16 | 26 | 16.3% | 4:30 |
| 6 | 挑战 | 11×17 | 32 | 17.1% | 5:00 |
| 7 | 挑战 | 11×19 | 37 | 17.7% | 5:30 |
| 8 | 困难 | 12×20 | 46 | 19.2% | 5:30 |
| 9 | 困难 | 12×22 | 53 | 20.1% | 6:00 |
| 10 | 困难 | 12×24 | 60 | 20.8% | 6:30 |

- 第 8 关起限时增幅明显小于盘面增幅——困难档的时间压力收紧。
- 数值为初版设定，完成后经试玩可调（集中在 levels.ts 一处配置）。

### 关卡推进

- **逐关解锁**：通过第 N 关解锁第 N+1 关；第 1 关默认解锁。
- 已通关卡可随时重玩刷成绩。

## 5. 游戏规则与判定

- 首次点击才开始倒计时。
- **失败**：倒计时归零（超时）或挖到雷。
- **通关**：所有非雷格全部揭开（插旗与否不影响判定）。
- 通关用时 = 限时 − 剩余时间；若小于该关历史最好成绩则更新并在结算中标注「新纪录」。
- 剩余雷数显示 = 总雷数 − 已插旗数（可为负，与经典扫雷一致）。

## 6. UI 设计

### 视觉

- **配色**：暖米白背景、白色卡片、深暖灰文字；无纯黑纯白、无高饱和色。三档点缀色：简单=鼠尾草绿、挑战=柔和琥珀、困难=柔和珊瑚。数字 1–8 用低饱和柔和色系区分。
- **形状**：卡片圆角 ≥ 20px、按钮圆角 ≥ 16px、棋盘格子约 30% 圆角；整体无棱角。
- **布局**：留白优先、间距充裕、不拥挤；单列纵向流式布局，天然适配手机竖屏与桌面窗口。

### 界面

1. **选关页**：标题 + 按三档分组的关卡卡片。锁定关灰显带锁图标；已通关显示最好成绩。
2. **游戏页**：顶栏（返回 ｜ 第 N 关 ｜ 剩余雷数 ｜ 倒计时）；棋盘居中；底部操作区（手机端为大号「挖开/插旗」模式切换 + 重开按钮，桌面端仅重开）。
3. **结算弹窗**：通关 → 本局用时、是否新纪录、「下一关」/「重玩」/「返回选关」；失败 → 失败原因（踩雷/超时）、「重试」/「返回选关」。

### 交互

- **桌面**：左键挖开、右键插旗、在已揭数字格上双键/双击快开（chord：旗数等于数字时揭开其余邻格）。
- **手机**：点按 = 执行当前模式（挖/旗）、长按 = 执行另一模式、底部按钮随时切换模式；数字格点按触发 chord。
- 触摸端禁用长按系统菜单、双击缩放等浏览器默认行为。

## 7. 数据持久化

- localStorage 存储：`{ unlockedLevel: number, bestTimes: Record<levelId, seconds> }`，带版本号字段便于将来迁移。
- 电脑版（Tauri WebView）与手机版（Safari PWA）各自独立保存，不做云同步（YAGNI）。
- localStorage 不可用时降级为内存态，并提示本次成绩不会保存。

## 8. 错误处理

| 场景 | 处理 |
|---|---|
| 无猜盘生成重试超过上限 | 继续尝试 + 显示轻量加载提示 + console 报警日志 |
| localStorage 读写异常 | 内存态降级 + 界面提示 |
| 存档数据损坏/版本不符 | 按损坏项重置为默认值，不影响其余数据 |

## 9. 测试策略

- **单元测试（Vitest）**：
  - board：连锁展开、插旗、chord、胜负判定、边界条件（角格/边格邻居数）。
  - solver：三层规则各自的正确判定用例（含 1-2-1 等经典型）。
  - generator：批量生成验证——每关多个随机种子，断言生成盘均可被求解器推完、首格 3×3 无雷、雷数正确，并统计生成耗时。
  - storage：读写、损坏数据恢复。
- **手动验收**：电脑浏览器、打包后的 .exe、iPhone Safari/PWA 实机各过一遍完整流程（新档 → 通关解锁 → 破纪录 → 失败重试）。
